# RAG（检索增强生成）

## Definition

RAG（Retrieval-Augmented Generation，检索增强生成）是一种在 LLM 推理时，通过从外部知识库中检索相关信息并注入提示上下文，从而提升生成质量的 AI 技术架构。

## Core Idea

LLM 的训练数据是静态快照——它只知道训练截止日期之前的信息，且无法区分"我知道"和"我猜的"。RAG 的核心洞见是：**与其让模型记住所有知识，不如在推理时给它一个可查询的外部知识库**。这相当于将 LLM 从"闭卷考试"变成"开卷考试"——模型不再依赖训练时记住的事实，而是从外部检索最新、最相关的信息作为推理依据。RAG 不改变模型本身，它改变的是模型获取信息的路径。

## Mechanism

RAG 的核心机制是一个四阶段管道，在 LLM 生成响应之前插入一个检索步骤：

**1. 知识外化（Encoding）**：将外部文档按语义单元切分（chunking），通过嵌入模型（embedding model）转换为高维语义向量，存入向量数据库。这一步将非结构化的文档知识转化为机器可计算的语义空间。

**2. 语义检索（Retrieval）**：用户查询同样被向量化，在向量数据库中执行近似最近邻（ANN）搜索。关键选择在于索引结构：HNSW 在中小规模数据集上以更高内存开销换取低延迟高召回；IVF 在大规模场景下更省内存。

**3. 上下文注入（Augmentation）**：将检索到的相关文档片段与原始用户提示拼接，形成增强提示。这一步是 RAG 区别于纯检索系统的关键——检索结果不直接返回给用户，而是作为 LLM 的"参考资料"进入推理过程。

**4. 响应生成（Generation）**：LLM 基于训练知识（参数化知识）和检索到的外部文档（非参数化知识）共同生成响应，并可附带来源引用。

管道质量具有复合效应：差的分块 → 差的嵌入 → 差的检索 → 差的响应。每个阶段必须独立验证。

## Applications

- **客服自动化**：聊天机器人通过 RAG 访问产品文档、政策文件和客户历史，提供来源可引用的准确回答
- **企业知识库问答**：组织内部文档（手册、FAQ、研究报告）作为外部知识源，员工以自然语言查询
- **实时信息场景**：连接新闻源、社交媒体或实时数据流的 RAG 系统可提供超出 LLM 训练截止日期的最新信息
- **专业领域应用**：法律、医疗等需要精确引用来源的领域，RAG 的归因能力比纯生成模型更可信

## Limitations

- **检索质量上限**：RAG 的生成质量不会超过检索质量。如果相关文档没有被检索到，LLM 就无法利用它们——而这取决于分块策略、嵌入模型和索引结构的选择
- **不适合全局推理**：RAG 擅长"找到相关段落并基于此回答"，但不擅长需要跨多文档综合推理的复杂查询
- **增加推理延迟**：检索步骤在 LLM 推理之前增加了额外的计算开销。在语音对话等低延迟场景中，管道中每个环节的延迟都会叠加
- **不改变模型能力边界**：RAG 不是微调的替代品。它改变了模型可访问的信息，但不改变模型的推理能力、风格或领域理解深度
- **知识库维护成本**：外部文档需要持续更新和重新嵌入。过时的外部知识会导致检索出的"权威信息"实际上是错误信息
- **噪音风险**：检索相关性不足时，注入的文档片段成为噪音而非有用上下文，反而降低响应质量

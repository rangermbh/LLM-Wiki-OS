# Knowledge Distillation Protocol（知识蒸馏协议）

本协议定义如何将原始采集内容蒸馏为长期知识概念页面。蒸馏位于 `/ingest` 管道中的 Routing 和 Concept Creation 之间。

Version: 1.0.0

---

## 1. 核心区分：文章摘要 vs 知识概念

| 维度 | 文章摘要（Article Summary） | 知识概念（Knowledge Concept） |
|------|---------------------------|------------------------------|
| 结构 | 跟随来源文章的顺序 | 按概念逻辑组织 |
| 语言 | 可能保留原文措辞 | 使用知识系统自身的语言 |
| 视角 | 代表来源作者的观点 | 代表系统对该概念的**理解** |
| 完整性 | 试图覆盖来源中的所有内容 | 提取对知识系统有价值的部分 |
| 稳定性 | 绑定到特定文章 | 设计为可随理解加深而演进 |
| 连接性 | 可能仅引用来源 | 与 Wiki 中其他概念交叉链接 |
| 营销语言 | 可能包含宣传性表述 | 剥离营销，仅保留技术实质 |

知识概念页面回答的是：**"我们的知识系统对这个概念的理解是什么？"** ——而不是："这篇文章说了什么？"

---

## 2. Concept Page 必须包含的结构

每个 Concept 页面必须包含以下区域：

### Definition（定义）

一句话捕获概念的本质。不是词典定义——而是将该概念置于知识系统所在领域中的定位。

### Core Idea（核心思想）

中心洞见。这个概念为什么重要？它解决了什么问题？2-3 句话，让没有读过来源文章的人也能理解。

### Mechanism（工作机制）

概念如何运作。内部逻辑、架构或流程。足够具体，以区别于相邻概念。

### Relationships（关系）

该概念与 Wiki 中其他概念的连接。每条关系应说明**关联的性质**，而非仅仅罗列名称。

### Applications（应用）

概念在实践中的应用场景。真实用例，非假设性可能。

### Limitations（局限）

该概念**不能**做什么。在什么情况下会失效。它解决不了什么问题。**此区域至关重要**——它防止知识变成教条。

### Sources（来源）

指向采集文件的可追溯链接。知识来源的证据。

### Notes（笔记）

开放性问题、计划中的连接、维护 TODO。

## 3. 蒸馏规则

### 必须做

- 提取概念的内在逻辑和机制
- 通过 wikilink 连接到 Wiki 中已有的概念
- 使用知识系统自身的语言（本仓库以中文为主）
- 在英文技术术语比中文翻译更精确的地方保留英文原词
- 保持来源可追溯性（`captured_from`、`sources`）

### 禁止做

| 禁止行为 | 说明 |
|----------|------|
| 原文顺序复制 | 不得按来源文章的结构和顺序组织页面 |
| 营销语言 | 剥离宣传性表述、厂商特定的热情推荐 |
| 观点堆积 | 不得将来源文章的观点不经综合地罗列 |
| 无关背景 | 不得包含来源中与概念无关的背景信息 |
| 来源崇拜 | 不得将来源文章视为概念的唯一权威 |

## 4. 蒸馏流程

### Step 1: 识别概念

采集文件是关于**哪一个**概念的？如果涉及多个概念，识别主要的那个。

### Step 2: 提取核心机制

内在逻辑是什么？如何运作？组件是什么，它们如何交互？

### Step 3: 识别关系

与 Wiki 中已有的哪些概念相关？揭示了什么新的关系？

### Step 4: 识别边界

该概念**不能**解释什么？在什么情况下失效？公认的局限性是什么？

### Step 5: 综合撰写

将页面作为知识系统对该概念的**理解**来写——而非来源文章的摘要。来源文章是**证据**，不是**模板**。

---

## 5. 质量检查清单

保存 Concept 页面前，逐项验证：

- [ ] Definition 是一句话，且将该概念置于领域之中
- [ ] Core Idea 无需阅读来源即可理解
- [ ] Mechanism 足够具体，能区别于相邻概念
- [ ] Relationships 说明了关联性质，而非仅罗列名称
- [ ] Limitations 存在且有实质内容
- [ ] 来源可追溯性已保留
- [ ] 无营销语言或厂商热情残留
- [ ] 页面结构未跟随来源文章结构
- [ ] 所有 wikilink 解析到已有文件或已记录为 planned
